# Reasoning Mode Comparison (思考模式对比)

本示例展示了如何使用 Hy3 API 的 `reasoning_effort` 参数控制模型的思考强度，并通过流式传输解决深度思考模式下的输出截断问题。

## 1. 核心逻辑
- **思考模式开关**：通过 `extra_body` 传入 `reasoning_effort` 参数控制。
  - `"none"`：关闭思考模式，响应速度最快，适用于简单问答。
  - `"high"`：开启深度思考，模型会先进行逻辑推演（Chain of Thought），回答更准确但耗时更长。
- **流式传输（关键）**：在非流式调用下，`high` 模式因思考过程过长常被截断。本示例采用 `stream=True`，实时接收 `reasoning_content`（思考过程）和 `content`（最终回答），确保完整输出。

## 2. 请求参数解析
- `model`: "hy3-preview" (指定使用的模型)
- `messages`: 对话历史数组。
  - `role`: "user" (用户输入)
  - `content`: "证明√2是无理数，要求逻辑严谨，步骤清晰"
- `extra_body`: Hy3 特有参数载体。
  - `reasoning_effort`: 控制思考强度 ("none" 或 "high")。
- `stream`: `True` (开启流式传输，解决深度思考输出截断问题)。
- `temperature`: 0.3 (控制随机性)。
- `max_tokens`: 8192 (为防止思考过程耗尽额度，设置较大的上限)。

## 3. 响应解析 (Response Parsing)
流式响应下，需区分两种增量内容：
- **思考过程**：`chunk.choices[0].delta.reasoning_content`
  - 仅在 `reasoning_effort="high"` 时出现。
  - 包含模型的逻辑推演、自我纠错等内部过程。
- **最终回答**：`chunk.choices[0].delta.content`
  - 模型基于思考过程生成的最终结果。
- **性能指标**：
  - `首字耗时 (TTFT)`：从发起请求到接收到第一个数据块的时间（深度思考模式下此值较长）。
  - `总耗时`：从发起请求到流式传输结束的时间。

## 5. 问题解决记录 (Troubleshooting)
**现象**：初始版本使用非流式调用并设置 `max_tokens=2048`，开启 `high` 模式后输出被截断（仅返回约500字符）。
**原因**：深度思考模式下，模型将大量 Token 预算分配给了后台思考过程，或直接受限于非流式接口的单次返回包大小限制。
**解决方案**：
1. 启用 `stream=True` 进行流式接收。
2. 监听 `delta.reasoning_content` 字段以获取完整的思考链条。
3. 适当调大 `max_tokens` 以支持长思考链。
