# Error Handling & Retry Mechanism (错误重试机制)

本示例演示了如何为 Hy3 API 调用实现健壮的错误重试机制，涵盖超时、限流及网络异常等常见场景，并采用**指数退避（Exponential Backoff）**策略优化重试频率，避免服务雪崩。

## 1. 核心逻辑
- **异常捕获**：精准捕获 OpenAI SDK 定义的特定异常，而非通用的 `Exception`（除兜底外）。
- **指数退避重试**：遇到可重试错误时，等待时间随重试次数指数级增长（`base_delay * (2 ** attempt)`），降低对服务端的压力。
- **失败兜底**：达到最大重试次数后，若仍失败则抛出异常终止程序，防止无限阻塞。

## 2. 请求参数解析
- `client` 配置：
  - `api_key`: 鉴权凭证（示例中使用的是测试Key）。
  - `base_url`: API 接入地址。
  - `timeout`: 10秒（单次请求最长等待时间，超时将触发 `APITimeoutError`）。
- `call_with_retry` 函数参数：
  - `max_retries`: 最大重试次数（不含首次请求）。
  - `base_delay`: 初始重试延迟基数（秒）。
- `messages`: 固定请求内容，用于测试算法生成能力。

## 3. 响应与异常处理解析
代码针对以下三种常见 API 错误进行了针对性处理：

| 异常类型 | 错误含义 | 处理策略 |
| :--- | :--- | :--- |
| `RateLimitError` | 触发速率限制 (HTTP 429) | 记录日志，按指数退避等待后重试。 |
| `APITimeoutError` | 请求超时 | 记录日志，按指数退避等待后重试。 |
| `APIConnectionError` | 网络连接失败 | 记录日志，按指数退避等待后重试。 |
| `Exception` (兜底) | 未知错误 (如认证失败) | **立即抛出，不重试**（如 401 认证错误重试无意义）。 |

## 5. 问题解决记录 (Troubleshooting)
**现象**：在测试 401 错误时，日志显示为“未知错误”而非具体的 `AuthenticationError`。
**原因**：当前使用的 `openai` Python 库版本或腾讯 TokenHub 网关的返回格式，可能未被 SDK 映射为标准的认证异常类，而是作为通用 API Error 返回。
**解决方案**：
代码中使用了 `except Exception as e` 作为兜底，虽然捕获了错误，但在生产环境中，建议结合 `response.status_code` 进一步细化判断（例如判断状态码是否为 401），以便进行更精准的告警或逻辑处理（如自动刷新 Token）。
