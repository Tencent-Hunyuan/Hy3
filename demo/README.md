# Hy3 RAG 端到端 Demo 录制指南

本目录用于存放提交到 rhinobird2026 issue 的演示素材。本文档给出**录制工具建议**、**两段 demo 的分步脚本**、**文件命名规范**以及**提交方式**。

> Issue 硬性要求：**至少 2 个端到端 demo 流程** + **≤2 分钟视频或 GIF**。
> 当前项目是「最干净的样子」（空知识库），录制前请先把样例文档放回（见下文「0. 录制前准备」）。

---

## 0. 录制前准备（重要）

应用当前知识库为空。录制前需要把 1~2 个样例文档放回去，二选一：

**方式 A：用 UI 上传（演示最自然，推荐用于 Demo 1）**
1. 启动应用：`cd hy3-rag && .venv\Scripts\python.exe run.py`
2. 浏览器打开 http://127.0.0.1:8766
3. 直接把文件拖进上传区（支持 PDF/DOCX/MD/TXT/代码/JSON 等 14 种格式）

**方式 B：用助手脚本一键入库简易样例（省事，适合反复录制）**

> ⚠️ 应用**只在「上传」时入库**，不会自动扫描 `data/uploads/` 里的文件。
> 所以只把文件拷进去、再重启应用是**不够的**——必须走一次上传接口。

简易样例文档位于 `demo/sample/`（`Hy3模型简介.md`、`Hy3常见问题.md`），简洁、适合 2 分钟演示。

**先启动应用**，再运行助手脚本（它会复制这 2 个样例文档并通过 API 真正入库）：

```bash
# 终端 1：启动应用
.venv\Scripts\python.exe run.py        # 打开 http://127.0.0.1:8766

# 终端 2：入库简易样例文档
.venv\Scripts\python.exe demo\prepare_demo_data.py
```

> 若想换成你自己的材料：直接把文件拖进 UI（方式 A），或替换 `demo/sample/` 里的文件后重跑脚本。

脚本会输出每个文档的 chunk 数；`/api/health` 应显示 `documents:3`（或更多）。

## 1. 录制工具建议（任选其一）

| 工具 | 平台 | 优点 | 输出 |
|------|------|------|------|
| **ShareX** | Windows | 免费、内置 GIF 压缩、可截窗口 | `.gif` |
| **OBS Studio** | Win/Mac/Linux | 高质量、可录 mp4 | `.mp4` |
| **浏览器插件**（如 "Screen Recorder"） | 跨平台 | 无需安装、直接录标签页 | `.webm`/`.mp4` |
| **macOS 自带** `Cmd+Shift+5` | Mac | 系统原生 | `.mov` |

**格式与体积建议**
- 优先导出 **GIF**（评审一眼能看，无需下载播放器）。
- 若 GIF 体积 > 5MB，改导出 **MP4**（≤2 分钟、720p 通常 < 3MB）。
- 录制区域选**应用窗口本身**（浏览器里 http://127.0.0.1:8766 那一页），不要录整个桌面。
- 窗口宽度建议 **1280px**，保证回答文字清晰可读。

## 2. Demo 1 —— 上传 → 检索 → 流式问答（核心能力展示）

**目标**：展示"多文档检索增强 + Hy3 流式生成 + 来源引用"。

**分步脚本（照着录）**
1. 打开 http://127.0.0.1:8766 ，页面显示空知识库（`文档 0`）。
2. 拖入 `Hy3模型简介.md` → 等待「已入库」提示（角标变为 `文档 1`）。
3.（可选）再拖入 `Hy3常见问题.md` → 角标 `文档 2`，体现"多文档"。（也可直接跑助手脚本一次性入库两份）
4. 在提问框输入：**「Hy3 是什么模型？它有什么核心特点？用 3 个要点概括。」**
5. 展示：回答**逐字流式**出现，底部出现「引用来源：Hy3模型简介.md」。
6. 再问一个跨文档的问题：**「如何调用 Hy3？支持多长的上下文？」** → 展示引用同时指向两份文档。

**录制要点**
- 开头 2 秒给一个「上传前空库」的镜头，对比上传后变化，更有说服力。
- 把光标停在「引用来源」区域停留 1 秒，让评审看清来源标注。
- 总时长控制在 **60~90 秒**。

**保存为**：`demo/demo1-upload-qa.mp4`（已录制）

## 3. Demo 2 —— 文件夹限定检索 + 多轮对话记忆

**目标**：展示"文件夹分类管理"和"对话历史保存/恢复"（应用的差异化能力）。

**分步脚本（照着录）**
1. 打开左上角菜单（☰）→ 点「📁 文档管理」→ 在弹窗的「文件夹」区点 **「+ 文件夹」** → 输入 `测试` → 确认。左侧侧边栏出现 `📁 测试` 标签。
2. 在弹窗的「文档 · 编辑组别」区，找到 `Hy3常见问题.md` 行 → 它的「组别」下拉选 `📁 测试`，文档即归入该文件夹。
3. 点 `📁 测试` 标签（激活该文件夹）→ 文档列表只剩 `Hy3常见问题.md`；此时提问框检索范围被限定在此文件夹内。
4. 提问：**「Hy3 支持多长的上下文？推理模式有什么用？」**
   → 展示只在 `测试` 文件夹内检索、Hy3 给出带引用的回答（与 Demo 1 的全库检索形成对比）。
5. 新建一个会话（「+ 新对话」），切换回去，再点「恢复历史」，**展示上一轮对话被完整恢复**（证明记忆持久化）。
6. （可选收尾）把 `Hy3常见问题.md` 的下拉改回「未分类」，或点 `📁 测试` 标签上的 `×` 删除文件夹——注意删除文件夹后文档会**自动回到「未分类」**，不会丢失。

**录制要点**
- 强调「选定文件夹后，检索范围被限定」这一点（和 Demo 1 的全库检索形成对比）。
- 恢复历史时把时间戳/标题露出来，证明是持久化存储而非内存。
- 总时长控制在 **60~90 秒**。

**保存为**：`demo/demo2-folder-memory.mp4`（已录制）

## 4. 文件命名规范

```
demo/
├── demo1-upload-qa.mp4        # 上传 → 检索 → 问答
├── demo2-folder-memory.mp4    # 文件夹限定 + 多轮记忆
├── README.md                  # 本文件
└── CHECKLIST.md               # issue 要求对照清单
```

> 命名统一 `demoN-<场景>.mp4`，方便在 PR 描述里逐一引用。

---

## 5. 提交方式

录完后把素材纳入仓库并提交：

```bash
git add demo/*.mp4
git commit -m "demo: add end-to-end walkthrough (upload-qa + folder-memory)"
git push
```

然后在 PR 描述里补充 demo 链接（可用 GitHub 渲染的相对路径，如
`demo/demo1-upload-qa.mp4`），或直接附上仓库链接。
