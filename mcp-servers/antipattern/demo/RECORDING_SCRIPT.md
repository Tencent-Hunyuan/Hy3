# Demo 视频录制脚本（CodeBuddy 操作）

> 预计时长：2-3 分钟
> 工具：OBS / 任何录屏软件
> 分辨率：1920x1080，确保 CodeBuddy 面板清晰可读
> 前置：CodeBuddy 已配置好 antipattern MCP Server 且状态为已连接

---

## 录制前准备

1. 打开 CodeBuddy，确认右侧 MCP 面板中 `antipattern` 显示 4 tools、状态为启用
2. 编辑器字体调到 16px 以上（Settings → Editor → Font Size）
3. 关闭所有无关通知（微信、钉钉、系统通知）
4. 关闭无关标签页，只留一个空白对话窗口
5. OBS 框选 CodeBuddy 窗口，确认录制分辨率 1920x1080
6. 先试录 5 秒确认画面清晰、文字可读

---

## 开场（10秒）

**画面**：CodeBuddy MCP 面板，展示 antipattern 已连接。

**操作**：
- 鼠标悬停在 antipattern 条目上，让观众看到 "4 tools" 字样
- 可以点开看一下 tool 列表（challenge_design / remix_paradigm / stress_test_orthodoxy / escalate）

**字幕/旁白（可选）**：
"这是 AntiPattern——一个基于 Hy3 的叛逆设计顾问 MCP Server，已经接入 CodeBuddy。"

---

## 第一幕：challenge_design（60秒）

**操作**：在 CodeBuddy 对话框中输入以下内容，然后发送：

```
我们团队准备启动一个新的内部后台管理系统项目，技术选型初步定的是：
- 前端框架：React 18
- 状态管理：Redux Toolkit
- UI 组件库：Ant Design 5
- 布局：经典的左侧边栏树形导航 + 右侧表格/表单列表
- 预计 30+ 页面，主要是 CRUD 表格、审批流、数据看板

团队 4 个人，都是 React 熟手但没用过 Redux Toolkit。
项目周期 3 个月，甲方是国企，要求"看起来像正经管理系统"。

请用 antipattern 的 challenge_design 工具对这个方案发起挑战，
领域选 ui，强度 4。我想看看这套"标准答案"里有没有什么我没意识到的坑。
```

**等待输出**（大约 20-40 秒，Hy3 深度推理中）。

**输出展示要点**（输出完成后，缓慢滚动展示）：
- 停在「你的假设（我闻到了）」3 秒 —— 让观众看到假设被点破
- 停在「质疑（带刺的）」5 秒 —— 这是最有戏剧性的部分，比喻金句在这里
- 快速滚过「反方案」—— 让观众知道有具体替代方案即可
- 停在「可行性自评（诚实的）」3 秒 —— 展示"不是为了反而反"
- 停在底部「本次思维武器：xxx」2 秒 —— 展示策略元信息

**注意**：如果输出过程中 CodeBuddy 做了二次总结/评论，也一起展示——这是"AntiPattern 开炮 + 宿主做裁判"的双层效果。

---

## 第二幕：stress_test_orthodoxy（40秒）

**操作**：新开一轮对话（或继续），输入：

```
最近团队里在争论架构选型。CTO 坚持要把现有单体拆成微服务，
理由是"业界都这么做，方便扩展"。但我们的实际情况是：
- 团队只有 5 个后端开发
- 目前已经拆了 30 个微服务（大部分是 CTO 主导拆的）
- 部署频率大概每周 2 次
- 没有专职运维，开发自己兼
- 服务间调用链最长 7 跳，排查问题经常要翻 5 个日志系统

请用 antipattern 的 stress_test_orthodoxy 工具，
压力测试"微服务优于单体架构"这条共识，
把我们的场景作为 context 传入。我想知道这条共识在我们的条件下到底成不成立。
```

**等待输出**。

**输出展示要点**：
- 停在「反方」部分 5 秒 —— 展示极端但严密的反驳（"5个园丁照看30座温室"之类的比喻）
- 停在「裁判裁决」3 秒 —— 展示适用边界判断
- 停在「你的立场建议」2 秒

---

## 第三幕：escalate 加码（30秒）

**操作**：继续同一轮对话，输入：

```
刚才 challenge_design 给的反方案还是太保守了，
什么"日志驱动轻栈"、"先埋点后选型"——这些我都能想到。

请用 antipattern 的 escalate 工具，把上一轮的完整输出传入，
强度拉到 5，方向是"彻底推翻重来，从第一性原理出发"。

我想知道：如果完全忘掉"后台管理系统"这个品类，
从零开始思考"4个人怎么最高效地服务30个内部页面的用户"，
答案会不会根本不是一个"系统"？
```

（CodeBuddy 有上下文记忆，会自动把上一轮输出传给 escalate tool）

**等待输出**。

**输出展示要点**：
- 停在「上一轮太温和了」2 秒 —— 展示自我批评
- 停在「更疯的反方案」5 秒 —— 这是高潮
- 停在「疯狂边界」2 秒 —— 展示诚实标注

---

## 收尾（10秒）

**操作**：
- 切回 MCP 面板，展示 4 个 tool 列表
- 或者在对话框打一行："AntiPattern — 你的最佳实践，是我的异端审判对象。"

**结束录制。**

## 后期处理

1. 裁剪头尾空白（ffmpeg 或剪映）
2. 压缩到 50MB 以内：
   ```bash
   ffmpeg -i demo_raw.mp4 -c:v libx264 -crf 28 -preset fast -c:a aac demo_final.mp4
   ```
3. 如果超过 GitHub 100MB 限制，上传到 B站/YouTube 取链接
4. 文件命名：`demo/antipattern_demo.mp4`

---

## 备选：如果某次输出质量不好

- 策略抽取有随机性，每次输出不完全一样
- 多录 2-3 次，挑比喻最精彩、结构最完整的那条
- 如果 Hy3 响应太慢（>60秒），可以后期加速等待部分（2x）
- 如果输出有错别字或格式问题，重录即可
